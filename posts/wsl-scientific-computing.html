Why I Still Recommend WSL: A Hands-On Record of Setting Up Scientific Computing on Windows 为什么我最终还是更推荐 WSL：一次在 Windows 上配置科学计算环境的折腾实录

Wen Chen | Apr 15, 2026 |

I originally set out to prove that "you don't really need WSL—Windows can handle it just fine." But after going through the whole process, I ended up understanding even better why WSL is worth recommending.

我本来是想证明"其实没必要 WSL，Windows 也能搞定"。结果认真走了一遍之后，反而更理解了为什么 WSL 值得推荐。

The Origin: A Question That Started All the Tinkering 起因：从一个问题引发的折腾

I've always used WSL for development—for a practical reason: I need a Linux environment for scientific computing, but I don't want to give up Windows entirely (e.g., all the games on Steam).

我一直用 WSL 做开发——原因很实际：我需要 Linux 环境做科学计算，但又不想完全放弃 Windows 的东西（比如 Steam 上各种游戏）。

After coming to the US, I noticed that students kept asking the same question: "Why do we need WSL? Can't we just set up a scientific computing environment directly on Windows?" This question piqued my curiosity. I'd always used WSL, but was it because Windows genuinely couldn't do it, or because I'd never seriously tried? I decided to tinker through it myself and find a convincing answer: Can you actually set up a decent scientific computing environment on pure Windows?

到了美国以后，我发现一个问题：学生们总问一个同样的问题——"为什么要装 WSL？直接在 Windows 上配科学计算环境不行吗？"这个想法戳到了我的好奇心。我一直用 WSL，但到底是 Windows 真的不行，还是我没有真正试过把它配好？我决定自己折腾一遍，找一个有说服力的答案：到底能不能在纯 Windows 上，配出一套像样的科学计算环境？

Python is actually not hard. Whether it's Anaconda, Miniconda, or venv + pip, they're all mature enough on Windows. The real challenges are:

Python 其实不难。无论是 Anaconda、Miniconda 还是 venv + pip，在 Windows 上都已经足够成熟。真正难的，是：

  • GNU toolchain: gcc, gfortran
  • GNU 工具链：gcc、gfortran
  • BLAS/LAPACK, e.g. OpenBLAS
  • BLAS/LAPACK，比如 OpenBLAS
  • Intel compilers: icx, ifx
  • Intel 编译器：icx、ifx
  • Intel oneAPI, MKL
  • Intel oneAPI、MKL
  • Large scientific computing libraries like PETSc and SLEPc that call into underlying compilers, BLAS, and MPI
  • 以及像 PETSc、SLEPc 这种会调用底层编译器、BLAS、MPI 的大型科学计算库

I thought this would be at most a one-day project. After finishing, I reached my real conclusion:

我以为这最多是个"花一天时间就能配好"的工程。结果折腾完之后，我得出了真正的结论：

It's not that Windows can't do scientific computing. It's that once you've actually gone through the whole process, you see clearly: what WSL saves you isn't just a few installation steps—it's a massive amount of energy that shouldn't be spent on environment friction.

不是 Windows 不能做科学计算。而是当你真正试过一遍后，你会更清楚地看到：WSL 省下来的不只是几个安装步骤，而是大量不该消耗在环境摩擦上的精力。

Why Can't You Just Install Compilers in PowerShell? 为什么不能直接在 PowerShell 里装编译器？

This is probably the first thought many Windows users have: I already have PowerShell, so why can't I just apt install gcc like on Linux and start working?

这个问题可能是很多 Windows 用户最先冒出来的念头：我都有 PowerShell 了，为什么不能像 Linux 上 apt install gcc 那样，直接装个编译器就开始干活？

The reason lies in a fundamental difference in design philosophy between Windows and Unix/Linux when it comes to development toolchains.

原因在于 Windows 和 Unix/Linux 在开发工具链的设计哲学上有根本性的区别。

On Linux, the system comes with a package manager (apt, dnf, pacman...). Compilers, header files, and libraries are all installed through a unified package management system—install them and they just work. Paths, environment variables, and dependencies are all handled automatically. You type gcc and it's gcc—no need to worry about where your headers are or whether the linker can find the libraries.

在 Linux 上，系统自带包管理器（apt、dnf、pacman……），编译器、头文件、链接库都通过统一的包管理系统安装，装完就能用——路径、环境变量、依赖关系全部自动处理好。你敲 gcc 就是 gcc，不需要额外操心"我的头文件在哪""链接器能不能找到库"。

Windows takes an entirely different approach. It doesn't have a system-level development package manager. Compilers (whether Microsoft's MSVC or GNU's gcc) require you to find the installer yourself and configure environment variables manually. More critically, Windows' native C/C++ toolchain is MSVC (Visual Studio's compiler), not gcc/gfortran. MSVC and the GNU toolchain differ in many details—header file paths, library file formats, linking methods, command-line arguments—they're all different.

Windows 的设计思路完全不同。它没有一个系统级的开发工具包管理器。编译器（无论是微软的 MSVC 还是 GNU 的 gcc）都需要自己去找安装包，自己配环境变量。更关键的是，Windows 原生的 C/C++ 开发工具链是 MSVC（Visual Studio 的编译器），而不是 gcc/gfortran。MSVC 和 GNU 工具链在很多细节上——头文件路径、库文件格式、链接方式、命令行参数——都不一样。

So if you want to use the GNU toolchain (gcc/gfortran) on Windows, you can't just "install it in PowerShell." You need an extra layer that provides the GNU compilers, corresponding headers, libraries, and an environment that makes it all work together.

所以，如果你想在 Windows 上用 gcc/gfortran 这套 GNU 工具链，你不是"在 PowerShell 里装一下"就能解决的。你需要一个额外的层，来提供 GNU 编译器、对应的头文件、库文件，以及让这一切能正常工作的环境。

This is exactly why Cygwin and MSYS2 exist.

这就是 Cygwin 和 MSYS2 存在的原因。

Cygwin: Simulating a Unix-Like System on Windows Cygwin：在 Windows 上模拟一个类 Unix 系统

Cygwin's core approach is to simulate a traditional Unix/Linux runtime environment on Windows as closely as possible. It provides its own runtime library (cygwin1.dll) that "translates" Unix-world filesystem interfaces, process management, shell behavior, and more into operations Windows can understand. This way, many programs that originally only ran on Linux can be compiled and run on Windows with minimal changes. It has a strong Unix flavor—many legacy tools and build systems feel like they're "still on Linux."

Cygwin 的核心思路是：在 Windows 上尽量模拟出一个传统 Unix/Linux 系统的运行环境。它提供自己的一套运行时库（cygwin1.dll），把 Unix 世界里的文件系统接口、进程管理、shell 行为等等都"翻译"成 Windows 能理解的操作。这样一来，很多原本只能在 Linux 上跑的程序，可以不怎么改就在 Windows 上编译运行。Unix 味道很浓，很多老工具、老构建系统在里面会觉得"还在 Linux"。

MSYS2: A Native GNU/MinGW Development Environment on Windows MSYS2：在 Windows 上提供 GNU/MinGW 原生开发环境

MSYS2 takes a different approach. While it also provides a bash shell and Unix tools, its emphasis is on providing a modern, practical GNU/MinGW development environment where compiled programs are Windows-native executables that don't depend on an extra compatibility layer. It ships with the pacman package manager (the same one Arch Linux uses), and tools like gcc, gfortran, cmake, pkg-config, and OpenBLAS can all be installed with a single command.

MSYS2 的思路不一样。它虽然也有 bash shell 和 Unix 工具，但更强调的是：提供一套现代、实用的 GNU/MinGW 开发环境，编译出来的程序是 Windows 原生可执行文件，不依赖额外的兼容层。它自带 pacman 包管理器（和 Arch Linux 同款），gcc、gfortran、cmake、pkg-config、OpenBLAS 这些东西都可以一条命令装好。

The key difference: programs compiled with Cygwin depend on cygwin1.dll (the compatibility layer)—they're more like "Unix programs running on Windows." MSYS2/MinGW compiles pure Windows-native programs that need no extra runtime.

两者的关键区别：Cygwin 编译出的程序依赖 cygwin1.dll 这个兼容层，更像"在 Windows 上运行的 Unix 程序"；MSYS2/MinGW 编译出的是纯 Windows 原生程序，不需要额外运行时。

Why I Started with Cygwin 我为什么一开始先折腾了 Cygwin

Because my goal at the time was to install PETSc/SLEPc and try connecting them with Intel oneAPI.

因为我当时的目标就是装 PETSc/SLEPc，并且试着把它们跟 Intel oneAPI 连起来。

I initially chose Cygwin over MSYS2 because the PETSc official documentation explicitly recommends Cygwin as the build environment in its Windows installation guide. Since upstream recommends it, that's where I started.

我一开始选 Cygwin 而不是 MSYS2，是因为 PETSc 官方文档里，在 Windows 安装指南中明确推荐了 Cygwin 作为构建环境。既然上游推荐，那就先照着来。

But the Cygwin installation experience itself was already uncomfortable. It's not a command-line install—it's a GUI installer where you have to manually search through a very long package list and check off each package you need (gcc, gfortran, make, various libraries...). The library names all look similar, and you're entirely on your own to know what to install. The gap compared to a single apt install command on Linux is huge.

但 Cygwin 的安装体验本身就已经让我感到不太舒服。它不是命令行安装，而是一个 GUI 安装器——你需要在一个很长的包列表里手动搜索、勾选你需要的每一个包（gcc、gfortran、make、各种库……）。这个界面的库名字又都很类似，全靠你自己知道该装什么。跟 Linux 上 apt install 一条命令解决的体验差距非常大。

But what actually made me abandon this path wasn't just Cygwin's installation experience—it was the problems I ran into later when trying to connect PETSc/SLEPc with Intel oneAPI. Here's what actually happened: I got oneAPI installed, Cygwin was installed too, and I spent considerable time dealing with the integration between the two systems, even trying symlinks. But despite all that, PETSc/SLEPc still couldn't compile reliably.

不过真正让我放弃这条路的，不仅是 Cygwin 本身的安装体验，而是后面把 PETSc/SLEPc 和 Intel oneAPI 连起来时遇到的问题。我的实际进展是：oneAPI 能装上，Cygwin 也装上了；我还花了不少时间处理两套系统的衔接，连软链接都试过。但即便这样，PETSc/SLEPc 还是没能稳定编译成功。

So for me, this wasn't "theoretically impossible"—it was "not practical enough given my environment and time budget." Maybe a pure GNU route would have gone more smoothly; but at least from this hands-on experience, what truly blocked me was the continuous friction between different environment layers. I ultimately chose to abandon this hybrid approach.

所以这件事对我来说，不是"理论上绝对不可能"，而是"在我当时那套环境和时间成本下，不够可行"。也许如果只走纯 GNU 路线会更顺一些；但至少从这次实践看，真正卡住我的，确实是不同环境层之间持续不断的摩擦。最终我选择放弃这条混合路线。

This was a key turning point in my thinking: when a development environment requires you to first understand how three or four different environment systems mesh together, it's already unsuitable as an onboarding solution for most people.

这是我观念转变的一个关键点：当一个开发环境需要你先理解三四套环境系统如何互相咬合时，它就已经不适合作为大多数人的入门方案了。

Compromise: Two Completely Separate Routes 妥协：分成两条彻底分离的路线

Later I gave up the idea of "mixing everything together" and split into two separate routes.

后来我放弃了"所有东西混在一起"的想法，改成两条路线分开走。

The reason is simple: on Windows, the GNU toolchain and Intel compilers/oneAPI are two fundamentally different systems. The former is closer to the Unix/GNU world; the latter deeply depends on the Windows native development environment—Visual Studio, Windows SDK, oneAPI's own environment scripts. If you force them into a single unified environment, what you get isn't "elegant unification" but "a hybrid where neither side is comfortable."

原因很简单：在 Windows 上，GNU 工具链和 Intel 编译器/oneAPI 这两套体系的思路完全不一样。前者更接近 Unix/GNU 世界，后者则深度依赖 Windows 原生开发环境——Visual Studio、Windows SDK、oneAPI 自己的环境脚本。如果硬要把它们揉成一套统一环境，最后得到的不是"优雅的统一"，而是"谁都不太舒服的混合物"。

So my final approach was:

所以我最后的做法是：

  • Route A: GNU toolchain on MSYS2
  • 路线 A： MSYS2 上的 GNU 工具链
  • Route B: PowerShell + Visual Studio + Intel oneAPI
  • 路线 B： PowerShell + Visual Studio + Intel oneAPI

I stopped pursuing "one environment to solve all problems" and accepted reality: on Windows, the more stable approach to scientific computing is often not unification, but isolation.

我不再追求"一套环境解决所有问题"，而是接受现实：Windows 上做科学计算，很多时候更稳的办法不是统一，而是隔离。

Route A: GNU Toolchain on MSYS2 路线 A：MSYS2 上的 GNU 工具链

This route handles gcc, gfortran, OpenBLAS, and pkg-config. The goal is clear: prepare a relatively stable GNU scientific computing environment on Windows, at least capable of common C/Fortran + OpenBLAS work.

这条线专门负责 gcc、gfortran、OpenBLAS、pkg-config。目标很明确：在 Windows 上准备一条相对稳定的 GNU 科学计算环境，至少能完成 C/Fortran + OpenBLAS 这一类常见工作。

I ended up not sticking with Cygwin and switched to MSYS2 because: MSYS2 is better suited for native GNU development on Windows, with more convenient package management. gcc, gfortran, OpenBLAS, and pkg-config are more straightforward to install. It's more like a "I'm just running the GNU toolchain on Windows" path, rather than adding a heavy Unix compatibility layer.

我最后没有继续坚持 Cygwin，而是转向 MSYS2，原因是：MSYS2 更适合做 Windows 上的 GNU 原生开发，包管理更方便，gcc、gfortran、OpenBLAS、pkg-config 装起来更直接。它更像是一条"我就是在 Windows 上跑 GNU 工具链"的路，而不是额外叠一层很重的类 Unix 兼容环境。

Installation Steps 具体安装步骤

Step 1: Install MSYS2 from the command line. In an admin PowerShell:

第一步：用命令行安装 MSYS2。 在管理员 PowerShell 里执行： 

winget install -e --id MSYS2.MSYS2 --accept-package-agreements --accept-source-agreements

After installation, assuming the default path is C:\msys64, verify:

安装完成后，假设默认路径是 C:\msys64，可以先检查： 

Test-Path C:\msys64
Test-Path C:\msys64\usr\bin\bash.exe

Step 2: Update MSYS2 from PowerShell.

第二步：在 PowerShell 里更新 MSYS2。 

C:\msys64\usr\bin\bash.exe -lc "pacman -Syu --noconfirm"
C:\msys64\usr\bin\bash.exe -lc "pacman -Syu --noconfirm"

Running it twice is intentional—after a major MSYS2 update, it often needs a second pass to refresh the system package state.

这里连跑两次，是因为 MSYS2 第一次大更新后，经常需要再次刷新系统包状态。

Step 3: Install the GNU toolchain and OpenBLAS.

第三步：安装 GNU 工具链和 OpenBLAS。 

C:\msys64\usr\bin\bash.exe -lc "pacman -S --needed --noconfirm \
  mingw-w64-ucrt-x86_64-gcc \
  mingw-w64-ucrt-x86_64-gcc-fortran \
  mingw-w64-ucrt-x86_64-openblas \
  mingw-w64-ucrt-x86_64-pkgconf \
  make cmake"

Here I chose the UCRT64 environment; the actual usage later is centered around C:\msys64\ucrt64\bin.

这里我选的是 UCRT64 环境，后面实际使用时也围绕 C:\msys64\ucrt64\bin 来配置。

Step 4: Connect the GNU environment to PowerShell. If you want to call gcc, gfortran, and pkg-config directly from a regular PowerShell, you need to configure PATH and PKG_CONFIG_PATH.

第四步：把 GNU 环境接到 PowerShell。 如果想在普通 PowerShell 里直接调用 gcc、gfortran、pkg-config，需要配置 PATH 和 PKG_CONFIG_PATH。

Temporary test:

临时测试： 

$env:Path = "C:\msys64\ucrt64\bin;" + $env:Path
$env:PKG_CONFIG_PATH = "C:\msys64\ucrt64\lib\pkgconfig"

Persistent user environment variables:

写成持久的用户环境变量： 

[Environment]::SetEnvironmentVariable(
  "Path",
  "C:\msys64\ucrt64\bin;" + [Environment]::GetEnvironmentVariable("Path","User"),
  "User"
)
[Environment]::SetEnvironmentVariable(
  "PKG_CONFIG_PATH",
  "C:\msys64\ucrt64\lib\pkgconfig",
  "User"
)

Then reopen PowerShell.

然后重开 PowerShell。

Step 5: Verify the GNU environment.

第五步：检查 GNU 环境是否正常。 

where.exe gcc
where.exe gfortran
where.exe pkg-config

gcc --version
gfortran --version
pkg-config --libs openblas

If everything is working, you should see the paths for gcc.exe, gfortran.exe, and pkg-config.exe, along with OpenBLAS linker flags.

如果一切正常，你应该能看到 gcc.exe、gfortran.exe、pkg-config.exe 的路径，以及 OpenBLAS 的链接参数。

Result 结果

This route worked. gcc works, gfortran works, OpenBLAS can be found via pkg-config, and both C and Fortran programs can link and run. For the goal of "I want a GNU scientific computing environment on Windows," the MSYS2 route is viable.

这条线最后跑通了。gcc 正常，gfortran 正常，OpenBLAS 能通过 pkg-config 找到，C 和 Fortran 程序都能链接运行。从"我想在 Windows 上有一套 GNU 科学计算环境"这个目标来说，MSYS2 这条线是可行的。

Route B: PowerShell + Intel oneAPI 路线 B：PowerShell + Intel oneAPI

The other route stays entirely native Windows: icx, ifx, Intel oneAPI, MKL, Intel MPI.

另一条路线完全不走 MSYS2，坚持 Windows 原生：icx、ifx、Intel oneAPI、MKL、Intel MPI。

Why It Can't Be Mixed into MSYS2 为什么不能混进 MSYS2

Because Intel oneAPI depends on the Windows native development environment—Visual Studio Build Tools, Windows SDK, oneAPI's own setvars.bat. If you force this into MSYS2/bash, you'll easily run into shell syntax inconsistencies, incomplete environment variable stacking, and path resolution issues. So I moved the Intel route entirely back to PowerShell.

因为 Intel oneAPI 依赖的是 Windows 原生开发环境——Visual Studio Build Tools、Windows SDK、oneAPI 自己的 setvars.bat。如果把这套东西强行塞进 MSYS2/bash 里，很容易出现 shell 语法不一致、环境变量没叠完整、路径解析出问题。所以我把 Intel 这条线彻底放回 PowerShell。

Installation Steps 具体安装步骤

Step 1: Install Visual Studio Build Tools (command line). This step can't be skipped. Install the C++ toolchain and Windows SDK in an admin PowerShell:

第一步：安装 Visual Studio Build Tools（命令行）。 这一步不能省。建议在管理员 PowerShell 里直接安装 C++ 工具链和 Windows SDK： 

winget install -e --id Microsoft.VisualStudio.2022.BuildTools `
  --accept-package-agreements --accept-source-agreements `
  --override "--quiet --wait --norestart --nocache --add Microsoft.VisualStudio.Workload.VCTools --add Microsoft.VisualStudio.Component.Windows11SDK.22621"

If you prefer the Windows 10 SDK, replace the last component with Microsoft.VisualStudio.Component.Windows10SDK.19041.

如果你更想用 Windows 10 SDK，可以把上面最后一个组件替换为 Microsoft.VisualStudio.Component.Windows10SDK.19041。

After installation, check that the VS developer shell exists:

安装完成后先检查 VS 开发者 shell 是否存在： 

Test-Path "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\Common7\Tools\Launch-VsDevShell.ps1"

Proceed only when it returns True.

返回 True 再继续后面的 oneAPI 环境加载步骤。

Step 2: Install Intel oneAPI. After installation you get icx, ifx, MKL, and Intel MPI (if you selected the corresponding components). But don't assume it's ready to use right away.

第二步：安装 Intel oneAPI。 安装之后你会得到 icx、ifx、MKL，以及 Intel MPI（如果选了对应组件）。但千万不要以为装完就能直接用。

Step 3: Load the environment correctly—this is the most critical step.

第三步：正确加载环境——这是最关键的一步。

Don't try running icx in a plain PowerShell. The correct approach is to first load the Visual Studio Developer Shell, then load the oneAPI environment.

不要在普通 PowerShell 里裸跑 icx。正确方式是先加载 Visual Studio Developer Shell，再加载 oneAPI 环境。

First execute:

先执行：

& "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\Common7\Tools\Launch-VsDevShell.ps1" -Arch amd64 -HostArch amd64

Then execute:

再执行：

cmd /c '"C:\Program Files (x86)\Intel\oneAPI\setvars.bat" intel64 >nul && set' | ForEach-Object {
    if ($_ -match '^(.*?)=(.*)$') {
        [System.Environment]::SetEnvironmentVariable($matches[1], $matches[2], 'Process')
    }
}

After these two steps, verify:

这两步做完后，检查：

where.exe icx
where.exe ifx
Get-ChildItem Env:MKLROOT

Only at this point are you truly in a usable Intel environment.

这时候才算真正进入了 Intel 那套可用环境。

A Pitfall: stdio.h Not Found 一个坑：stdio.h 找不到

I had clearly installed Visual Studio Build Tools, Windows SDK, and Intel oneAPI. In theory, icx should have no trouble compiling the simplest C program. I wrote this:

我明明已经装好了 Visual Studio Build Tools、Windows SDK、Intel oneAPI。按理说，icx 编译一个最简单的 C 程序不该有问题。我写了这么一个文件： 

#include <stdio.h>

int main(void) {
    printf("ICX OK\n");
    return 0;
}

But when I compiled it, I got: stdio.h: file not found.

结果一编译，报错：stdio.h: file not found。

This was extremely confusing at first because I had clearly installed the SDK. After investigating, I found:

这件事一开始非常迷惑，因为我明明装了 SDK。后来排查发现：

  • Visual Studio Build Tools: installed
  • Visual Studio Build Tools 在
  • Windows SDK: installed
  • Windows SDK 在
  • The stdio.h file itself: on disk
  • stdio.h 文件本身也在磁盘上
  • But the current shell's INCLUDE environment variable didn't include the MSVC/UCRT/Windows SDK include paths
  • 但当前 shell 的 INCLUDE 环境变量里，并没有带上 MSVC/UCRT/Windows SDK 那几层 include 路径

In other words: you must load the Visual Studio developer environment first, then load the oneAPI toolchain; if the order is wrong, the compilation environment is incomplete.

也就是说：你要先加载 Visual Studio 开发者环境，再加载 oneAPI 工具链；顺序不对，编译环境就不完整。

Only after I loaded Visual Studio Developer Shell first and then stacked oneAPI setvars on top did icx finally find stdio.h.

直到我先加载 Visual Studio Developer Shell，再叠加 oneAPI setvars，icx 才终于能找到 stdio.h。

This was the first time I very clearly realized: the scientific computing environment on Windows isn't one environment—it's a stack of several environments layered on top of each other.

这件事让我第一次非常明确地意识到：Windows 上的科学计算环境，不是一个环境，而是几套环境的叠加。

Testing Intel Compilers 测试 Intel 编译器

Once the environment is loaded correctly, test icx:

环境加载对了之后，测试 icx：

@'
#include <stdio.h>

int main(void) {
    printf("ICX OK\n");
    return 0;
}
'@ | Set-Content test_icx.c

icx test_icx.c /Fe:test_icx.exe
.\test_icx.exe

Intel MPI: Even More Troublesome Than MKL Intel MPI：比 MKL 还更麻烦

MKL worked, but Intel MPI's wrapper had quirks with argument splicing and invocation in the Windows/PowerShell scenario. There was even an incident where copying the PowerShell prompt and error messages together into the terminal caused the errors to snowball.

MKL 测通了，但 Intel MPI 的 wrapper 在 Windows/PowerShell 场景下会出现参数拼接和调用上的怪问题。中间还发生过把 PowerShell 提示符和错误信息一起复制进终端、让报错滚雪球式扩大的事故。

It's not that Intel MPI can't work, but rather: when your development environment already depends on multiple shells, multiple toolchains, and multiple wrapper layers, any tiny operational mistake quickly escalates the complexity. And this complexity is never about algorithms or numerics—it's purely about the environment.

不是说 Intel MPI 不能用，而是：当你的开发环境已经依赖多个 shell、多套工具链、多层 wrapper 时，任何一个很小的操作失误都会迅速把问题复杂化。而且这种复杂化常常不是算法问题，也不是数值问题，是纯环境问题。

Even After Setup, Daily Use Is Still Inconvenient 即使配好了，日常使用仍然很不方便

If the article stopped at "look, I succeeded," that would actually be misleading. The reality is: even after both routes are set up, daily use is still full of friction.

如果文章停在"看，我成功了"，那其实是在误导人。现实是：即使两条路线都配通了，日常使用依然充满摩擦。

You Must Always Know Which Terminal You're In 你必须时刻知道自己在哪个终端里

I made a classic blunder myself: I was actually in MSYS2's bash terminal but instinctively typed a PowerShell command:

我中间就干过一件典型的蠢事：当时其实在 MSYS2 的 bash 终端里，结果下意识输入了一条 PowerShell 命令： 

& "C:\Program Files (x86)\Microsoft Visual Studio\..."

Bash immediately threw a syntax error—& "..." is PowerShell syntax.

bash 当然立刻报语法错——& "..." 是 PowerShell 语法。

On the surface it's a beginner mistake, but the underlying issue is: when doing scientific computing on Windows, you easily find yourself facing multiple contexts simultaneously—bash, PowerShell, cmd, VS developer shell, oneAPI environment shell. The moment you forget which environment you're in, commands go wrong instantly.

表面看是低级错误，本质上却是：你在 Windows 上做科学计算时，很容易同时面对 bash、PowerShell、cmd、VS 开发者 shell、oneAPI 环境 shell 等多个上下文。一旦忘了自己在哪个环境里，命令立刻错位。

In WSL, most of the time you're facing a single, unified shell context.

而在 WSL 里，大多数时候你面对的是一个统一的 shell 语境。

The Same Command, Completely Different Syntax Across Shells 同一条命令，不同 shell 语法完全不同

  • echo (...) > file works in cmd but throws errors in PowerShell
  • cmd 能用的 echo (...) > file，到 PowerShell 直接报错
  • PowerShell's here-string syntax can't be used in bash
  • PowerShell 的 here-string 写法，bash 里不能照搬
  • & is a call operator in PowerShell but has different semantics in bash
  • & 在 PowerShell 是调用符，在 bash 里是别的语义

You don't just need to know scientific computing—you have to constantly switch your "command language brain." For scientific computing users, this is almost pure overhead.

你不仅要懂科学计算，还要不断切换自己的"命令语言脑回路"。对科学计算用户来说，这几乎是纯粹的额外负担。

You End Up Maintaining "Two Worlds That Can't Mix" 最终你不得不维护"两套不能混的世界"

My most stable approach was to accept reality:

我最稳定的做法是接受现实：

  • GNU route stays GNU
  • GNU 线就是 GNU 线
  • Intel route stays Intel
  • Intel 线就是 Intel 线
  • Don't mix them
  • 不要混
  • Each has its own startup script, shell, and paths
  • 各自有一键启动脚本，各用各的 shell、各用各的路径

The final workable solution isn't "one unified Windows scientific computing environment"—it's two isolated environments, switched via scripts. It works, but it's not elegant.

最终可行的方案，不是"一个统一的 Windows 科学计算环境"，而是两套彼此隔离的环境，靠脚本切换。能用，但不优雅。

So Why I Still Recommend WSL 所以我为什么最终还是推荐 WSL

After all this tinkering, I didn't conclude that "Windows absolutely can't do it." A more accurate statement is:

折腾到这里，我并没有得出"Windows 完全不行"的结论。更准确的说法是：

Windows can certainly host a scientific computing environment, but it demands too much from the user.

Windows 当然能配科学计算环境，但它对用户的要求太高了。

You need to understand: the differences between MSYS2 and Cygwin, between the Windows native toolchain and the GNU toolchain, the relationships between VS Build Tools / Windows SDK / oneAPI, the differences between PowerShell / cmd / bash, and the behaviors of wrappers / paths / environment variables / headers / linkers...

你需要理解：MSYS2 和 Cygwin 的差异、Windows 原生工具链和 GNU 工具链的差异、VS Build Tools / Windows SDK / oneAPI 之间的关系、PowerShell / cmd / bash 的差异、包装器 / 路径 / 环境变量 / 头文件 / 链接器行为……

All of these are worth learning, of course. But the problem is: most scientific computing users aren't here to study the Windows command-line ecosystem.

这些东西当然都值得学。但问题是：大多数科学计算用户不是来学习 Windows 命令行生态的。

What they actually want to do is:

他们真正想做的是：

  • Install PETSc, install SLEPc
  • 装 PETSc、装 SLEPc
  • Compile a numerical experiment
  • 编译一个数值实验
  • Run parallel programs
  • 运行并行程序
  • Do matrix operations, debug algorithms
  • 做矩阵运算、调试算法

Not spend days switching between different shells.

而不是花几天时间在不同 shell 之间切来切去。

This is exactly where WSL's value lies. It's not recommended because it's "cooler"—it's because it collapses a massive amount of environment complexity into:

WSL 的价值恰恰在这里。 它不是因为"更酷"才值得推荐，而是因为它把大量环境复杂度收敛成了：

  • A unified Linux user space
  • 一个统一的 Linux 用户空间
  • A unified set of shell conventions
  • 一套统一的 shell 习惯
  • A unified package management logic
  • 一套统一的软件包逻辑
  • A runtime environment that matches what scientific computing software natively expects
  • 一套更贴近科学计算软件原生预期的运行环境

Not to mention that 99% of scientific computing servers run Linux.

而且更不要说99%的科学计算服务器是 Linux 系统了。

The Windows native route is for: people already very familiar with the Windows development environment; people who clearly know why they want to stay Windows-native; people with the patience to maintain multiple toolchains.

Windows 原生路线适合： 已经很熟悉 Windows 开发环境的人；明确知道自己为什么要坚持 Windows 原生的人；有耐心维护多套工具链的人。

WSL is better for: people who want to minimize environment friction; people who want students to get to the actual scientific computing faster; people who want consistency with Linux servers/clusters; people who want fewer "wrong terminal, wrong path, broken wrapper" pitfalls.

WSL 更适合： 想减少环境摩擦的人；想让学生更快进入科学计算正题的人；想跟 Linux 服务器/集群环境保持一致的人；想少踩"终端错了、路径错了、wrapper 坏了"这类坑的人。

I originally set out to prove that "you don't really need WSL—Windows can handle it just fine."

我最开始是想证明"其实没必要 WSL，Windows 也能搞定"。

But my real takeaway was: precisely because I seriously went through the Windows path, I now understand better why WSL is worth recommending.

但最后我的真正收获是：正因为我认真把 Windows 这条路走了一遍，我才更理解为什么 WSL 值得推荐。

Not because Windows can't do it, but because—once you've truly worked through all the complex environments—you realize: what WSL saves you isn't just installation steps. It's the massive amount of energy that should never have been spent on environment friction.

不是因为 Windows 不行，而是因为——当你真正把复杂环境都摸透以后会发现：WSL 省下来的，不只是安装步骤，而是大量本不该消耗在环境摩擦上的精力。
